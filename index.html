Pre-production — connectors built, schema v3 (access scoring, co-occurrence) in progress

Locigram is a self-hosted memory layer for AI assistants, accessible via MCP and REST. It ingests content from email, tickets, chat, and financial systems; extracts facts into addressable memory units (locigrams); and promotes high-confidence, repeatedly-accessed facts into truths via a decay-based scoring engine.

Storage is Postgres (structured) + Qdrant (vector). Deployment is Docker Compose or Kubernetes. LLM roles (embed, extract, summary) each point at any OpenAI-compatible endpoint. Connectors activate automatically from environment variables — none are required to start.

Data Model

TermDefinition
Locigram A single memory unit. One fact, event, or observation. Immutable once stored. Has content, locus, entities, confidence, source, and access scoring metadata.
Truth A high-confidence fact synthesized from multiple reinforcing locigrams by the truth engine. Has its own confidence score. Source locigrams go cold after promotion.
Palace One user or org's complete memory store. Isolated Postgres schema + Qdrant collection. Multiple palaces share a single deployment; separate K8s instances for hard isolation.
Locus A namespace for grouping and filtering recall. Examples: people/alice, business/acme, project/locigram, technical/firewall.
Connector A plugin that pulls from an external source, applies per-source noise filtering, and feeds the ingest pipeline. Activates at startup when its env vars are present.
Reference data Stable facts about things (devices, contracts, contacts). Flagged is_reference = true. Never processed by the truth engine. Superseded by creating a new record with the old one's expires_at set — history preserved, nothing deleted.

Truth Engine

Access scoring and temporal decay

Every locigram tracks recall frequency (access_count) and recency (last_accessed_at). A nightly sweep recomputes access_score for all non-reference locigrams:

access_score = access_count / (days_since_last_access + 1)decay_exponent
-- high recall frequency + recent access → high score -- never recalled → score approaches zero as age increases -- decay_exponent is configurable; default 1.5

Score thresholds determine tier transitions. Low-score locigrams in cold tier are sent through an LLM re-assessment pass to confirm they're garbage before expiration — this prevents false positives on rarely-accessed but legitimately important facts.

Hot
access_score above warm threshold. Full vector + full-text indexed. Returned first in recall.
Warm
Score dropped below hot threshold after inactivity. Still indexed; recall weight reduced.
Cold
Further decay, or source locigrams promoted into a Truth. LLM noise re-assessment runs here. Expire if confirmed junk; archive if still valid.
Truth
Synthesized from multiple reinforcing locigrams by the summary LLM. Carries its own confidence score. Source locigrams demoted to cold.

Co-occurrence clustering

The retrieval_events table records which locigram IDs are returned together per query. A weekly sweep unnests these ID arrays and counts pairwise co-occurrence frequency across all events. Pairs that appear together above a configurable threshold get cluster_candidate = true on both locigrams. The truth engine picks up cluster candidates and passes them to the summary LLM for merging into a single truth.

This is a better promotion signal than time-based logic alone: co-retrieval is direct evidence that two locigrams are conceptually related, regardless of when they were ingested or how similar their raw content is. Two records from different connectors, months apart, can be correctly identified as describing the same underlying fact.

Reference data

Locigrams with is_reference = true are excluded from truth engine processing and temporal decay. They represent stable facts about things that exist rather than events that happened.

reference_typeExamplesSuperseded when
network_deviceFirewall IP, switch hostname, VLAN IDNinjaOne sync detects a change
softwareApp version, license seat count, install stateNew version ingested
configurationPolicy settings, baselines, thresholdsChange detected in source
service_accountUsernames, roles, permissions (not credentials)Account deprovisioned
contractSLA terms, renewal dates, pricing tiersRenewal date passes
contactPerson name, phone, role, orgPerson leaves org or role changes

Superseding a reference record sets expires_at = now() on the old record and creates a new one. Records are never deleted — history is always recoverable.

Connectors

Connectors auto-register at startup when their required env vars are present. Each connector applies its own noise filter before any content reaches the pipeline or LLM.

Microsoft 365 — Email
Incremental sync via Graph API delta link. Filters: spam, junk folder, noreply senders, newsletters, OTP codes, calendar accept/decline messages, drafts, sent items. Requires Mail.Read + User.Read.All (Application permissions).
Microsoft 365 — Teams
Thread-based ingestion. Messages grouped by reply thread and submitted as a single unit after 2 hours of inactivity. Threads under 20 words are dropped. Filters: bot/webhook messages, system events, pure acknowledgements ("ok", "thanks", reactions). Requires ChannelMessage.Read.All, Team.ReadBasic.All, User.Read.All.
HaloPSA
Tickets ingested as knowledge. Assets and contracts ingested as reference data (is_reference = true). Internal-only test tickets, system-generated tickets, and closed tickets with no notes are skipped. Requires scopes: read:tickets, read:assets, read:contracts, read:clients.
NinjaOne
All device records ingested as reference data (is_reference = true, reference_type = network_device). Alerts ingested as knowledge events. Requires permissions: Monitoring, Management, Reporting.
Gmail
Incremental sync via historyId. Same noise filters as M365 email. OAuth2 refresh token required (one-time consent flow). Scope: gmail.readonly.
QuickBooks Online
Invoices, payments, vendor bills, time activities. LLM extraction is skipped — records are pre-classified by the connector directly from QBO's structured data. Dollar amounts are stored in metadata JSONB, not in content, to prevent LLM paraphrasing of financial figures. Invoices/bills are reference data; payments/time activities are knowledge events. Requires scope: com.intuit.quickbooks.accounting.
Webhook (always on)
Push any content directly via POST /api/webhook/ingest. No config required. Accepts sourceType: email, chat, ticket, device, invoice, payment, llm-session, note, manual, and others. Primary path for LLM session summaries and manual entries.
# Webhook example
curl -X POST http://localhost:3000/api/webhook/ingest \
  -H "Authorization: Bearer $API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "content": "Accudata firewall throwing alerts since 14:00. Blocked 203.0.113.42. Notified client.",
    "sourceType": "note",
    "metadata": { "client_id": "accudata" }
  }'

LLM Configuration

Three independently-configured roles. Each accepts any OpenAI-compatible endpoint. Omit *_KEY for local/unauthenticated endpoints. LOCIGRAM_SUMMARY_* falls back to the extract role if unset.

RoleWhat it doesURL varModel var

Extract Parses entities, locus, is_reference, confidence, and importance from raw content. Requires reliable structured JSON output. LOCIGRAM_EXTRACT_URL LOCIGRAM_EXTRACT_MODEL
Summary Truth promotion — synthesizes multiple locigrams into a single merged truth. Also used for LLM noise re-assessment on cold locigrams. LOCIGRAM_SUMMARY_URL LOCIGRAM_SUMMARY_MODEL

Architecture

TypeScript monorepo (Bun workspaces). Each package has its own declared dependencies — Bun does not hoist transitive deps.

@locigram/core
Shared types and Zod schemas. Source of truth for SourceType, Locus, and all cross-package interfaces.
@locigram/db
Drizzle schema + raw SQL CREATE TABLE IF NOT EXISTS migrations run at startup. No migration journal — idempotent by design.
@locigram/server
Hono REST API. MCP server (Streamable HTTP transport). Entry point for Docker image.
@locigram/pipeline
Ingest loop: receive raw content → LLM extraction (or skip if preClassified) → entity resolve → dedup check → write to Postgres + Qdrant.
@locigram/truth
Truth engine: promotion logic, decay sweep, co-occurrence cluster analysis, noise re-assessment.
@locigram/vector
Qdrant wrapper. One collection per palace. Point ID stored as embedding_id on the locigram.
@locigram/registry
Connector plugin registry (singleton). Scans env vars at startup, instantiates matching connectors.
@locigram/connector-microsoft365
M365 email (Graph delta sync) + Teams (thread-grouped, 2h cold threshold).
@locigram/connector-halopsa
HaloPSA tickets (knowledge), assets and contracts (reference).
@locigram/connector-ninjaone
NinjaOne devices (all reference) + alerts (knowledge).
@locigram/connector-gmail
Gmail via historyId incremental sync.
@locigram/connector-qbo
QuickBooks Online. PreClassified path — LLM extraction skipped, financial amounts in JSONB metadata.
@locigram/connector-webhook
Always-on push connector. Primary integration point for LLM session summaries.
